# CEP Phase 프롬프트 개선 리포트 — 2026-07-21

## 0. 개요

- **트리거**: 사용자 'CEP 생성 프롬프트 개선 요청'(2026-07-21). 신규 번들 검수가 아니라, **사용자가 직접 진단·요청한 methodology 축** 개선.
- **문제 성격**: grounding(할루시네이션·출처 근거)은 통과하는데 **CEP 이론(Category Entry Point 정의) 정합**이 아직 부분적이다. 두 축 중 grounding이 아니라 **methodology 축** 쪽 문제다.
- **축 상태**:
  - grounding 축 — 이번 회차 **신규 데이터 없음**. 기존 v1.2~1.4 규칙 유지(불변).
  - methodology 축 — 사용자 요청을 **일반 규칙(general=true)** 출처로 채택. 방법론 축 채택 규칙상 general은 반복 1건이라도 채택.
- **경계**: 프로덕션 프롬프트를 덮어쓰지 않고 **`미확정) v.X.Y.Z` 초안**만 생성. 실제 재생성·회귀 측정은 hubble/개발 파이프라인 소관(자동화 안 함).

## 3. 핵심 변경점 (클래스: 전부 methodology · 출처: 사용자 요청 2026-07-21 · general)

### Phase 4 v1.5.0 (crux)
- **M6 제품 편익·기능 배제** — situation은 상황·불편·감정·제약까지만. 기능·효과(자동비움·건조·완전 흡입 등)는 `kbf_hints`로 이동(삭제 아님).
- **M7 진입 장면만 + 사후 역추출** — 사용·구매 후 만족·검증 서사는 CEP 아님. 원문이 후기면 그 뒤의 '결핍의 첫 순간'을 역추출(단, quote 근거 필수).
- **M8 6W1H 하드 게이트** — Why+When 필수 + 나머지 2개↑, **근거 있는 축으로만** 충족. 미달 유닛은 카드 제외.
- **M9 압축 CEP + 상세 분리** — `cep`≈40자(회상 단서) / `context` 140~200byte(근거 디테일) / `rtb`(verbatim). **규칙 D 개정**.
- **M10 3축 평가 출력** — `evaluation`{market_potential, brand_fit, provability}(각 1~5, basis, estimated) + `cep_score`(곱). 시장성·적합성 데이터 없으면 `estimated:true`.
- **M11 페르소나 다양성** — 1인가구/맞벌이/반려동물/신체제약/육아/공간제약 등 진입 축 분산, 한 출처 과파생 금지.

### Phase 3 v1.3.0
- **P1 진입 순간 우선 채집** — 사후 만족·검증 후기는 그 뒤의 '결핍의 계기'를 함께 기록하고 사전/사후 **구분** 표기(단계 분리 규칙과 연동).
- **P2 진입 맥락 다양성** — 동일 상황·출처 과파생 방지, 근거 범위 내 페르소나 축 분산.

### Phase 5 v1.2.0
- **K1 기능·편익의 KBF 소유** — Phase4 M6로 CEP에서 뺀 기능·편익을 KBF가 정당하게 소유·정리. **단 v1.1.0 속성 그라운딩 불변** — 근거 있는 기능만 KBF로(창작 금지).

## 4. ⚠ 백엔드/다운스트림 영향 (승인 시 반영 필요)

Phase 4 v1.5.0은 **출력 스키마가 바뀐다**:
- **신규 필드**: `context`(문장형 상세), `evaluation`(3축 점수 + `cep_score` + `rank`), `w7`(6W1H 하드 최소 판정 대상).
- **의미 변경**: `cep`가 압축 문장(≈40자)으로 변경(길이 규칙 D 개정). 종전 `situation`(140~200byte)의 근거 디테일은 `context`가 승계.
- **파서/저장**: `context`·`evaluation` 파싱·저장 추가. `evaluation.market_potential`·`brand_fit`의 `estimated:true`(추정치)를 다운스트림에서 표기.
- **대시보드 검수기**: `cep`(압축)를 그라운딩 대상으로 삼으면 근거가 줄어 통과가 쉬워진다 — 상세 근거는 `context`/`rtb`에 있으니 검수 대상 필드 정의를 다시 볼 것.
- **하류 Phase**: Phase 5·5-1은 `card.cep`(압축) 입력과 호환된다. 다만 KBF/예시질문 품질을 생각하면 `context`를 함께 참조하도록 권장(선택). nanoIntents 제거(규칙 C)는 그대로 유지.
- Phase 5-1은 규칙을 안 바꿨지만 입력 `{{cep}}`가 압축 문장이 되니, **예시질문이 지나치게 축약된 맥락만 반영하지 않도록** 승인 뒤 `context` 병행 입력 여부만 정하길 권장.

## 5. 두 축 간 긴장 해소 (초안에 명시된 것 요약)

- **M6(기능 배제) vs 근거 보존** → 기능은 삭제가 아니라 `kbf_hints`로 **이동**(Phase 5 K1이 회수).
- **M7(역추출) vs grounding** → 역추출한 계기도 quote 근거 필수(결론부 비약 금지와 동일).
- **M9(압축) vs 규칙 D** → 디테일은 `context`/`rtb`로 이관(유실 아님).
- **M8(하드 6W1H) vs 할루시 금지** → 근거 있는 축으로만 최소치 충족, 미달이면 카드 수 축소(양보다 근거).
- **M10(3축 점수) vs 데이터 부재** → 시장성·적합성은 `estimated:true`로 표기(추정 명시).

## 6. 회귀 측정 안내 (자동 아님)

승인 → 개발이 프롬프트 반영 → 재생성 → 동일 검사기로 harvest → 본 스킬을 새 번들에 재실행하여 전후 비교:
- **methodology 지표(신규)**: ① 기능·편익 혼입 CEP 비율 ↓ ② 사후검증형 CEP 비율 ↓ ③ 6W1H 최소치 미달 비율 ↓ ④ `cep` 평균 길이 ≈40자 수렴 ⑤ 페르소나 축 분산도 ↑.
- **grounding 회귀 불변**: partial/vague/속성근거없음 플래그가 악화되지 않아야 함(특히 M9 압축이 근거를 과도하게 깎지 않는지, M6 이동이 KBF 그라운딩을 깨지 않는지).

## 8. 미결/승인 필요 사항

1. **스키마 변경 승인** — `context`·`evaluation` 필드 신설, `cep` 압축(§4). 백엔드 반영 범위 확정 필요.
2. **3축 점수 데이터 소스** — 시장성·브랜드 적합성에 쓸 데이터(검색량·브랜드 자산 등)가 있으면 `estimated:false`로 상향 가능. 없으면 추정 표기 유지.
3. **CEP 길이 목표** — 권장 40자(≤45)로 잡음. 팀 표준과 다르면 조정.
4. **Phase 5-1 입력** — 압축 `cep`만 줄지, `context` 병행 입력할지 결정.
