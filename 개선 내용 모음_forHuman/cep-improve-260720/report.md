## 2. 검수 산출물 (repropose)

- **재제안 대상:** 50개 프로젝트(warning/fail이 1건 이상). 49P보다 2개 늘었다 — 신규 #1371·#1372가 들어오고 #1373은 mismatch 0이라 빠졌다.
- **재활용 판정 결과 (52P):** 재사용 48 — `cep-improve-260720/repropose/`의 기존 산출물이 신규 번들의 mismatch seq와 정확히 맞아떨어졌다. 복사 0, 재생성 2(#1371, #1372).
- **재생성 배치:** 대상이 2개로 BATCH(6) 안에 들어와 한 번에 병렬 처리했다.
- **최종 검증:** 50개 모두 통과. `cep-improve-260720/repropose/*.json`의 mismatch_points 168개가 전부 입력 seq와 맞았다. 판단 기준은 파일이 실제로 있는지, JSON이 파싱되는지, seq가 일치하는지를 기계로 확인한 것이고, 서브에이전트가 돌려준 응답 텍스트는 근거로 삼지 않았다.
- 대시보드 반영: 새로 만든 `repropose/1371.json`, `repropose/1372.json`을 리포 루트의 대시보드용 `repropose/` 폴더에 복사한 뒤 `node gen-dashboard-data.js <52P 번들>`을 다시 돌렸다. `dashboard-data.js`와 `dashboard.html`의 캐시 버전도 갱신했고, 52개 프로젝트·520카드·repropose 50개가 제대로 반영됐는지 확인했다.

## 5. 회귀 측정 안내 (자동 아님 — 개발/hubble 파이프라인 소관)

### 성과 목표: 전체 정합 수치 ≥ 70%
- 정합 수치는 pass 비율이다(pass / 전체 카드). warning/fail은 비정합으로 본다.
- **49P 베이스라인:** pass 325/490 = 66.3%
- **52P 현재:** pass **352/520 = 67.7%**(의심+실패 168 = 32.3%). 신규 3P 중 #1373이 전체 pass라 비율이 조금 올랐지만 목표에는 아직 못 미친다.
- **목표:** pass **≥364/520 (70.0%)**. 지금보다 pass 12장을 더 돌려세워야 하고, 의심+실패는 168에서 156 이하로 내려가야 한다.

### 절차
1. `미확정` 초안 3개(§4, 변경 없음)와 사용자가 정한 4규칙(종결어미, 인용부호 제거, 나노인텐트 제거, 140~200byte)을 검토하고 승인한다.
2. 승인한 규칙을 개발이 프로덕션 Phase 3·4·5 프롬프트에 반영한다. 백엔드 정합도 맞춰야 하는데, 특히 nanoIntents를 주입하고 소비하는 경로를 없앤다.
3. 같은 대상으로 CEP를 다시 생성한다.
4. 같은 검사기로 harvest를 돌려 새 번들을 만든다.
5. 이 스킬을 새 번들에 다시 돌려 pass율(정합 수치)·source_grounding partial·플래그 빈도를 전후로 비교한다.
   - 1차 합격선은 pass율 70% 이상이다. 곁들여 볼 지표로는 source_grounding partial(현재 106)이 줄었는지, vague_expression과 KBF 속성 창작에 걸린 프로젝트 수가 줄었는지를 본다.
   - 70%에 못 미치면 남은 실패의 why를 다시 군집해 phase 규칙을 보강한다. 반복 채택 기준은 똑같이 적용한다.

### JP/EN
데이터와 원문이 KR이라 초안도 KR만 손봤다. JP/EN은 같은 취지(시점·단계 분리, 감정·체감 창작 금지, KBF 정성 속성 그라운딩)를 사람이 직접 반영해야 한다.
